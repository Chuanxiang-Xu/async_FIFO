# Learning Async FIFO

This document is a guided reading path for people who want to learn how an
asynchronous FIFO works, not only instantiate it as reusable IP.

If you only need the integration contract, read
[Interface and Timing](interface.md). If you want the module hierarchy first,
read [Architecture](architecture.md). If you want to start from a basic FIFO
and walk slowly through the waveform, read the
[step-by-step tutorial](tutorial.md). Then come back here and follow the
mechanism from problem statement to RTL.

The theoretical baseline is Clifford E. Cummings, *Simulation and Synthesis
Techniques for Asynchronous FIFO Design*, SNUG San Jose 2002
([technical-library entry](https://www.sunburst-design.com/papers/CummingsSNUG2002SJ_FIFO1.pdf)).
This repository uses the synchronized-pointer comparison style from that
family of designs, with wrappers kept outside the CDC core.

| Local binary read pointer | `rptr_bin` in [`rtl/core/rptr_empty.v`](../rtl/core/rptr_empty.v) |
| Extra pointer bit | `ADDR_WIDTH + 1` pointer width in [`rtl/core/async_fifo_core.v`](../rtl/core/async_fifo_core.v) |
| Binary-to-Gray pointer conversion | `wptr_gray_next` and `rptr_gray_next` |
| Write pointer crossing to read clock | [`rtl/core/sync_w2r.v`](../rtl/core/sync_w2r.v) |
| Read pointer crossing to write clock | [`rtl/core/sync_r2w.v`](../rtl/core/sync_r2w.v) |
| Empty generated in read domain | `rempty_next` in [`rtl/core/rptr_empty.v`](../rtl/core/rptr_empty.v) |
| Full generated in write domain | `wfull_next` in [`rtl/core/wptr_full.v`](../rtl/core/wptr_full.v) |
| Payload kept out of synchronizers | [`rtl/core/fifo_mem.v`](../rtl/core/fifo_mem.v) |

The important project-specific addition is `rd_valid`. The core RAM uses a
synchronous read template, so [`rtl/core/async_fifo_core.v`](../rtl/core/async_fifo_core.v)
registers `rd_valid` from `read_allow` to tell the user when `rd_data` belongs
to an accepted read.

## Terminology used in this guide

| Term | Meaning in this project |
|---|---|
| core | The equal-width FIFO implementation that owns CDC, memory, pointers, and local status |
| wrapper | A protocol or width adapter around the core; it should not change the CDC mechanism |
| payload | The data item stored in the FIFO; for stream mode this includes metadata such as `keep` and `last` |
| beat | One accepted interface transfer |
| occupancy | A local-domain used-count view, not a perfect global count |
| Gray pointer | A binary pointer converted to Gray code before crossing clock domains |

## 1. What problem does an async FIFO solve?

An asynchronous FIFO safely moves data between two unrelated clock domains:

```text
write clock domain                 read clock domain
------------------                 -----------------
wr_en, wr_data  --->  FIFO RAM  ---> rd_en, rd_data
wr_clk                 CDC          rd_clk
```

The two clocks may have different frequencies, phases, jitter, or even pause
independently. That means the write side cannot directly trust a multi-bit
counter generated by the read side, and the read side cannot directly trust a
multi-bit counter generated by the write side.

The FIFO solves this by separating the design into three ideas:

- payload data stays in dual-clock memory;
- each side owns its local binary pointer;
- only registered Gray-coded pointers cross clock domains through
  synchronizers.

In this repository, that mechanism lives in [`rtl/core/`](../rtl/core/). The
top-level [`rtl/async_fifo.v`](../rtl/async_fifo.v) is intentionally a small
public entry point.

## 2. Two local views, not one global state

An async FIFO does not have one perfect global occupancy register. Instead,
each clock domain has a local view:

| Domain | Owns | Receives through CDC | Generates |
|---|---|---|---|
| write | write pointer | synchronized read pointer | `full`, `almost_full`, `wr_used` |
| read | read pointer | synchronized write pointer | `empty`, `almost_empty`, `rd_used` |

This is why `full` belongs to the write clock domain and `empty` belongs to the
read clock domain. They are safe for local flow control, but they are not a
simultaneous global snapshot.

The design is deliberately conservative. When the other side makes progress,
the local side may learn about it a few cycles later because the pointer has to
pass through a synchronizer. This can make `full` or `empty` deassert later
than the physical RAM state changed, which is safe.

## 3. Why binary pointers are not synchronized directly

A FIFO pointer is naturally a binary counter:

```text
000 -> 001 -> 010 -> 011 -> 100 -> ...
```

The problem is that several bits may change at the same time. For example,
`011 -> 100` flips three bits. If this multi-bit bus is sampled by another
clock domain during the transition, the receiver might observe a mixed value
that was never a real pointer.

An async FIFO avoids this by converting the binary pointer to Gray code before
crossing domains:

```text
binary: 000 001 010 011 100 101 110 111
gray:   000 001 011 010 110 111 101 100
```

Only one Gray-code bit changes per increment. After synchronization, the
received pointer may be old or new, but it is much less likely to become an
unrelated arbitrary count. Physical timing constraints are still required so
the Gray bus reaches the first synchronizer stage with bounded skew.

Relevant RTL:

- [`rtl/core/wptr_full.v`](../rtl/core/wptr_full.v): write binary pointer and
  write Gray pointer
- [`rtl/core/rptr_empty.v`](../rtl/core/rptr_empty.v): read binary pointer and
  read Gray pointer
- [`rtl/core/sync_w2r.v`](../rtl/core/sync_w2r.v): write Gray pointer
  synchronized into the read domain
- [`rtl/core/sync_r2w.v`](../rtl/core/sync_r2w.v): read Gray pointer
  synchronized into the write domain

## 4. Why the pointer has one extra bit

For a FIFO with `2**ADDR_WIDTH` entries, the RAM address uses `ADDR_WIDTH`
bits, but the pointer uses `ADDR_WIDTH + 1` bits.

The low bits address the RAM. The extra high bit distinguishes two cases that
can otherwise look identical:

- pointers equal because the FIFO is empty;
- lower address bits equal after the writer has wrapped one lap ahead.

That extra wrap bit is what lets the design tell empty and full apart.

```text
ADDR_WIDTH = 3
RAM address bits = pointer[2:0]
wrap bit         = pointer[3]
```

## 5. Empty detection

The read domain owns the read pointer. It also receives a synchronized copy of
the write pointer.

The FIFO is empty when the next read pointer would equal the synchronized write
pointer:

```text
next read Gray pointer == synchronized write Gray pointer
```

Intuitively: after accepting this read-side action, there would be no unread
entries left from the read domain's point of view.

In the RTL, this logic is in
[`rtl/core/rptr_empty.v`](../rtl/core/rptr_empty.v).

## 6. Full detection

The write domain owns the write pointer. It also receives a synchronized copy
of the read pointer.

For the common Cummings-style async FIFO structure used here, the FIFO is full
when the next write Gray pointer catches the synchronized read pointer with the
two most significant bits inverted and the remaining bits equal.

Conceptually:

```text
next write pointer is exactly one FIFO depth ahead of synchronized read pointer
```

The Gray-code comparison looks slightly strange because the pointer includes
the extra wrap bit and is already converted to Gray code. The key idea is
simple: the writer must stop before it overwrites unread data.

In the RTL, this logic is in
[`rtl/core/wptr_full.v`](../rtl/core/wptr_full.v).

## 7. Where the data lives

Payload data does not pass through the two-flop pointer synchronizers. It stays
in a dual-clock RAM:

- write side writes `wr_data` into the RAM at the write address;
- read side reads `rd_data` from the RAM at the read address.

The pointers decide which addresses are safe to use. The RAM carries the actual
payload.

In this repository, [`rtl/core/fifo_mem.v`](../rtl/core/fifo_mem.v) models the
storage so synthesis tools can infer FPGA RAM resources.

## 8. Reset handling

The FIFO has separate active-low resets for the write and read domains:

- `wr_rstn` resets write-domain state;
- `rd_rstn` resets read-domain state.

Reset may assert asynchronously, but release must be synchronized to the local
clock. The helper
[`rtl/util/async_reset_sync.v`](../rtl/util/async_reset_sync.v) implements that
pattern.

Reset is destructive. Treat it as reinitializing the FIFO, not as preserving
queued data. During startup, both sides should complete coordinated reset
release before normal traffic begins.

## 9. Why constraints still matter

Two-flop synchronizers reduce metastability risk, but they do not replace
timing and CDC sign-off.

The intended CDC structure is:

```text
binary pointer -> Gray register -> first sync flop -> second sync flop
```

The physical implementation must keep the Gray-code bus well controlled from
the source Gray register bank to the first synchronizer stage. Otherwise,
different Gray bits could arrive with excessive skew and defeat the one-bit
change assumption at the sampling edge.

For Vivado, see
[`constraints/xilinx/async_fifo.xdc`](../constraints/xilinx/async_fifo.xdc) and
[CDC Constraints](cdc_constraints.md).

## 10. Why wrappers are outside the core

The core is intentionally equal-width and protocol-light. It owns the CDC
problem:

```text
async_fifo_core
├── memory
├── write pointer and full
├── read pointer and empty
└── Gray pointer synchronizers
```

Wrappers adapt interfaces around that core:

- [`async_fifo_width_conv`](../rtl/wrappers/async_fifo_width_conv.v) packs or
  splits words for different interface widths;
- [`async_fifo_stream`](../rtl/wrappers/async_fifo_stream.v) stores
  `{data, keep, last}` as one payload and adds ready/valid packet behavior.

Keeping wrappers outside the core makes the CDC learning path easier: first
understand the equal-width FIFO, then study how protocol adapters reuse it.

## Suggested reading order

1. Read the [step-by-step tutorial](tutorial.md) and run `make tutorial` to
   connect `wr_en`, `rd_en`, `full`, `empty`, and `rd_valid` with the waveform.
2. Read the small integration example in
   [`examples/basic_fifo/`](../examples/basic_fifo/).
3. Read [`rtl/async_fifo.v`](../rtl/async_fifo.v) to see the public entry
   point.
4. Read [`rtl/core/async_fifo_core.v`](../rtl/core/async_fifo_core.v) to see
   how the pieces connect.
5. Read [`rtl/core/wptr_full.v`](../rtl/core/wptr_full.v) and
   [`rtl/core/rptr_empty.v`](../rtl/core/rptr_empty.v).
6. Read [`rtl/core/sync_w2r.v`](../rtl/core/sync_w2r.v) and
   [`rtl/core/sync_r2w.v`](../rtl/core/sync_r2w.v).
7. Read [CDC Constraints](cdc_constraints.md) to connect the RTL idea to FPGA
   implementation.
8. Read the wrappers only after the equal-width core feels natural.

The mental model to keep while reading the RTL is:

```text
data path:    RAM, indexed by local binary pointers
control path: Gray pointers crossing clock domains
status path:  local flags computed from local pointer + synchronized remote pointer
```
